### 问题1：AI出题有很大的不确定性，有时候题目甚是错误的

目前的 prompt 设计存在几个明显薄弱点：

1. 题目领域没有任何约束
   - useCopilotAdditionalInstructions 只写了"准确性以 2025 年最新资料为准"，但完全没说题目必须围绕"太阳系/天文学"主题。
   - 模型可能出"哪一年发射 XX 探测器"这种它本身就不准的细节题（LLM 对具体年份/数字记忆容易混淆）。
2. 没有让模型对答案做内部校验
   - 要求 LLM 先生成 4 个选项再标 `correctAnswer` 时，它会自由发挥，并不一定先确定答案再构造干扰项；
   - 也没有写"四个选项必须互不相同；正确答案必须严格出现在选项中且唯一正确"这种校验规则。
3. 没有禁列
   - 没有禁掉那些大模型容易答错的题型（具体年份、具体距离/质量数值、最新探测任务进度等），结果就出来了它"自信地写错"的题。
4. `useCopilotReadable` 只暴露了 `questionList`，没有暴露"当前游戏阶段 / 还要出几题 / 难度梯度"等上下文，模型缺少出题节奏依据。

**解决方向**

- 在 useCopilotAdditionalInstructions 中显式锁定：

  - 题目主题：太阳系、行星、恒星、宇宙基础常识；
  - 出题流程：****先在心里确认答案** → 再围绕该答案构造 3 个合理干扰项 → 最后写 correctAnswer 必须等于 options 中之一**；
  - 题型偏好：优先概念题、特征题，避免具体数值/年份/最新进展；
  - 难度梯度：q1–q3 入门，q4–q6 中等，q7–q8 进阶。

- 在 `setQuestion` 这个 tool 的 `description` 里再次强调"correctAnswer 必须是 options 里的某个键，且四个选项互不相同"。

- 把"已经回答过的题目"的题干也单独传给模型作为去重 readable，避免出重复题。

  

### 问题2：用户选了答案后"有时没触发 setQuestion"

这是更典型的工具调用不稳定，看 onSelect 的实现就能定位到几个原因：



1. 发给 AI 的消息措辞太"对话化：

   ```
   content: `我选择了答案 ${option}，${
     args.question?.questionNumber !== 'q8' ? '请出下一题' : '游戏结束'
   }`
   ```

   这在 LLM 看来更像普通用户聊天。它完全可能选择

   只回一段文字反馈（"恭喜你答对了"），而不是调用 setQuestion。模型选不选 tool 是概率性决定的，措辞越像"聊天"，跳过 tool 的概率越大。

2. 指令里没有把"必须调用 setQuestion"写成强约束：

   - 现在写的是"用户每回答完一题…继续调用 setQuestion 进行下一轮出题"。
   - 但同一段又写了"如果用户提问的是已经回答过的问题，可以告诉用户答案" —— 这两条会让模型在面对"我选了 A"时陷入"是该回话还是该出题"的犹豫。

3. 题目编号是隐式的。模型要自己根据 `questionList` 推断"现在该出 q几"。一旦上下文长了，它经常推断错误，干脆用文本回复"下一题准备好了吗？"代替工具调用。

4. ****followUp: false 本身没问题（避免 tool 跑完后再多一段啰嗦的 LLM 文本），但配合上面 1、2 两点，会让"AI 直接发一段文本而不调用工具"这种情形看起来更突兀，因为它后面就再也不补出题动作了。

5. q8 → 游戏结束的措辞同样模糊。`gameOver` tool 是 `useCopilotAction`，没有 schema 强约束，AI 经常用文本说"游戏结束啦"而不调用 `gameOver`，结果 `gameState` 永远停在 q8。

**解决方向**

可以叠加做以下几件事：

1. 改写发给 AI 的消息，写成命令式 + 显式带题号：

   ```
   content:
     args.question?.questionNumber !== 'q8'
       ? `用户选择了 ${option}（题号 ${args.question.questionNumber}）。请立刻调用 setQuestion 出下一题（题号应为 ${nextNumber}），并在 aiFeedback 中告知本题正确答案是否为 ${args.question.correctAnswer}。`
       : `用户选择了 ${option}（题号 q8，最后一题）。请立刻调用 gameOver。`;
   ```

   带上题号 + "请立刻调用 X"会显著提高工具调用率。

2. 在 instructions 中加入显式触发规则：

   - "只要用户消息中包含'我选择了答案'或'请出下一题'，必须调用 setQuestion，禁止只用文字回复。"
   - "题号为 q8 的题被回答后，必须调用 gameOver，禁止只用文字宣布游戏结束。"

3. 把"下一个题号"作为 readable 暴露给模型，比让它自己根据 questionList 推断更稳：

   ```
   useCopilotReadable({ description: '下一个应出的题号', value: nextQuestionNumber });
   ```

4. 加一层兜底：onSelect 里 `setQuestionList` 之后启动一个 6–8 秒的超时，如果 `gameState` 还停留在原题号，就再次 `sendMessage`（带更强指令）触发一次。这个适合在 prompt 改完仍偶发漏调用时打底。

5. ****可选****：把 `gameOver` 的 description 写得更明确——"当 questionList 长度达到 8 或用户主动结束时必须调用"。
